# 项目开发规范（本仓库专用）

## 环境说明
- 框架：CodeIgniter 4.5.7
- PHP 版本：≥ 8.1（`public/index.php` 设定的最低版本）

## 注释与文档要求

1. **生成代码注释规范**
   - **数据查询来源**：明确标注数据来源（如数据库表名、API接口地址）。
   - **查询限制条件**：必须说明查询的状态限制（如 `status=1` 代表有效数据）。
   - **业务逻辑说明**：清晰描述关键业务逻辑和处理流程。
   - **算法实现思路**：对于复杂算法，需解释实现思路。

2. **注释格式规范**
   - **函数/方法**：使用标准文档格式（如 PHPDoc/JSDoc/JavaDoc）。
   - **行内注释**：关键代码行必须添加行内注释解释意图。
   - **变更标注**：业务规则变更时，必须在注释中标注变更原因。

3. **注释质量标准**
   - **拒绝模糊描述**：禁止使用"查询数据"等无意义描述，必须包含具体条件（如"查询当月有效订单"）。
   - **多表关联说明**：涉及多表关联查询时，必须说明关联关系和目的。
   - **状态流转标注**：涉及状态变更时，必须注明状态值及其业务含义。

4. **审核机制**
   - **提交前检查**：代码提交前需自检注释覆盖率。
   - **核心代码要求**：核心业务逻辑代码注释率要求 100%。
   - **合规性**：模糊、不准确的注释视为不合规，需返工修正。

## 代码风格
- 代码风格需与现有项目保持一致（对齐方式、命名习惯、异常与返回处理方式等）。
- 在已有模式下扩展，不随意引入新的风格或依赖。

## 前端开发规范
- **去插件化 (De-plugin)**
  - **原则**：凡是能用 CSS 解决的 UI 效果，**绝对不要**引入 jQuery 插件。
  - **例子**：表格固定列、自定义滚动条、简单的折叠面板、悬停效果 —— 全部用 CSS + Bootstrap Class。
- **jQuery 仅作“辅助”**
  - 只用 jQuery 处理 AJAX 请求（`$.post`）和简单的点击事件（`on('click')`）。
  - **禁止**用 jQuery 去负责复杂的界面布局计算。
- **拥抱 Bootstrap 5 原生特性**
  - Bootstrap 5 自带了很多强大的交互（模态框、下拉菜单、Tooltips），它们是用原生 JS 写的，性能更好，且不会和 CSS 打架。
  - **优先使用**它们，而不是去找第三方的 jQuery 插件。

## 分层职责（Controller / Service / Model）
- Controller
  - 负责路由入口、参数获取与校验，调用 Service 完成业务。
  - 视图加载：使用 `BaseController::showView([ 'view/path' ])`，默认自动包裹 `/common/head` 与 `/common/footer`；如不需要头尾可传 `autoHeader = false`。
  - 数据返回：业务错误/异常统一用 `Tools::ResponseMsg($st, $msg, $url = '', $data = '')` 输出 JSON；保持与现有控制器一致的返回方式（如 `showJmsg`）。
- Service
  - 承担业务逻辑与流程编排，组合多个 Model/外部接口，处理状态流转校验、事务等。
  - 处理业务逻辑的函数必须有中文注释，说明流程与入参含义。
  - Controller 不重复 Service 中的业务判断，保持职责单一。
- Model
  - 仅负责表映射、字段定义、基础查询（继承 CI4 `BaseModel` 的用法）。
  - 禁止在 Model 中加入业务流程，业务规则放到 Service。

## 视图与数据返回
- 视图：Controller 通过 `showView` 传入视图数组，可按需组合多个视图；默认会加载公共头尾。
- 数据返回：统一用 `Tools::ResponseMsg` 输出 JSON，或沿用项目现有封装的成功/失败提示，保持一致风格。

## 适用范围
- 本规范仅适用于本仓库。每次改动代码前需遵循以上要求。

## 工作流与执行规则（6A 简要要求）
- 全程中文回复。
- Align：先确认需求、识别不确定点，必要时提问，方案与项目架构/版本对齐。
- Architect：仅在需要时输出接口/数据结构设计，复用现有分层模式，不做过度设计。
- Atomize：内部拆分子任务，确保覆盖需求、依赖清晰。
- Approve：检查计划完整性、一致性、可行性、可测性后再实施。
- Automate：按子任务执行，代码遵守本规范和现有风格；敏感信息用 .env；若遇阻塞/不确定必须暂停并提问；默认不写测试，除非明确要求。
- Assess：自检实现是否满足需求/验收标准，与架构一致且兼容当前框架版本，若有未完成项需标注 TODO。

## 测试与数据库操作注意
- 写完代码应执行恰当的测试；若无法测试需在回复中说明原因。
- **禁止自动执行** 修改表结构、回滚数据库等高度敏感操作；如测试需要变更表结构或回滚数据，必须先征得确认后才能执行。
