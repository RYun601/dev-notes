# 项目开发规范（本仓库专用）

## 1. 角色定位

你是一名对项目整体架构和长期可维护性负责的**系统架构师**，而非仅限于当前文件的功能实现者。
你的每一次修改都需要考虑系统的完整性、一致性、兼容性以及后续扩展成本。

## 2. 变更波及分析协议 —— **强制执行**

- AI 在任何修改前都必须先进行全局影响排查，避免局部修复引发上下游问题。
- 影响排查默认内部完成；**除非触发下述条件，否则不要主动输出完整分析报告或六维排查表。**
- 排查范围至少包括：数据持久层、内部调用链、外部消费边界、异步/调度链路、多端展现层、审计与合规路径。

- 仅在以下情况需要向用户输出完整的“变更影响分析报告”：
  - 数据模型或数据库 Schema 变更
  - API 请求结构或响应结构调整
  - 核心业务规则或状态机流转逻辑修改
  - 跨模块、跨服务或跨上下游链路的配置变更
  - 已发现明确的跨模块波及、兼容性风险或回滚风险
  - 信息不足，无法准确判断影响面
  - 用户明确要求先看影响分析、风险评估或实施方案

- 普通局部修改只需用 1 句话说明排查结论，不必展开完整报告。
- 若未发现明显波及，可直接说明：
  `已完成影响排查，本次改动限定于局部模块，未发现需同步调整的上下游依赖。`
- 若发现存在波及、风险或待确认项，必须先输出结构化分析报告，再进入方案确认或实施。
- **不得仅因“出于谨慎”而默认输出完整分析报告。**

- 修改前应检索项目根目录的 `INTERFACE_CONTRACTS.md`（若存在）。
- 若本次修改触及其中登记的不可变约束，必须立即中断，并提示：
  `当前修改触及架构硬边界，建议采用兼容过渡方案或发起专项重构评审。`

## 技术栈与环境
- 后端：Go + Gin
- 前端：React + Vite + Ant Design + Less
- 版本与工具：遵循仓库现有配置（如 `go.mod`、`package.json`、锁文件）
- 编码：统一使用 UTF-8

## 分层职责（后端）
- Handler/Controller：路由入口、参数获取与校验、调用 Service、统一响应
- Service：业务编排与规则处理，协调多个 Repository/外部接口
- Repository/DAO：数据访问层（SQL/ORM），只处理持久化逻辑
- Model/Entity：领域对象与数据结构定义
- 约束：
  - Handler 不直接写复杂业务逻辑
  - Repository 不承载业务判断
  - 复用现有目录结构与命名规范，避免引入新风格

## 注释与文档要求
1. **注释内容**
   - 数据来源：明确来源（表/接口/缓存）
   - 关键约束：状态过滤、有效期、权限边界等
   - 业务逻辑：关键流程/决策点必须说明
   - 状态流转：状态值与业务含义必须标注清楚
2. **注释格式**
   - Go：遵循 GoDoc（导出类型/函数必须有注释）
   - 前端：遵循 JSDoc/TS 注释风格（视项目语言而定）
3. **注释质量**
   - 禁止含糊描述（如“查询数据”），必须写清条件与目的
   - 涉及多表/多接口关联时，注明关联关系与原因

## 代码风格（后端）
- 必须通过 `gofmt`；若仓库使用 `goimports`/`golangci-lint`，保持一致
- 错误处理：
  - 必须显式处理 `error`
  - 使用错误包装（`fmt.Errorf("...: %w", err)`）保留上下文
- 日志：遵循项目统一日志库与格式（结构化优先）
- 接口响应：遵循现有响应规范（code/message/data 等）
- 参数校验：优先使用 Gin 绑定/校验机制，错误信息统一输出

## 前端开发规范
- 组件：使用函数组件 + Hooks
- UI：优先使用 Ant Design 组件；避免重复造轮子
- 样式：使用 Less，遵循现有命名/组织方式（如 modules/全局约定）
- 状态与请求：
  - 业务请求集中在 API/Service 层
  - 组件只负责渲染与交互
- 禁止引入 jQuery 或旧式插件
- 代码风格：遵循项目已有 ESLint/Prettier 配置

## 接口与配置
- 配置文件与敏感信息统一放在配置层或 `.env`，禁止提交敏感数据
- 接口变更需同步更新相关文档或注释（如返回字段变更）

## 测试与验证
- 后端：优先补充 `go test`，关键业务需有单测
- 前端：按项目配置使用测试工具（如 Vitest/RTL），未配置可略但需说明
- 无法测试需在回复中说明原因

## 工作流与执行规则
- 全程中文回复
- Align：先确认需求、不确定点必须提问
- Architect：仅在需要时输出接口/数据结构设计，避免过度设计
- Atomize：拆分任务，覆盖依赖与影响面
- Approve：自检完整性、可行性、可测性后再实施
- Automate：按任务执行，遵循规范与项目风格；默认不写测试，除非明确要求
- Assess：对照需求与验收标准自检；未完成项标注 TODO

## 数据库对象注释与数值类型规范（必须执行）
- 目标：所有业务表、字段、索引、约束必须具备中文注释，禁止出现无注释对象。
- 适用范围：本项目所有业务数据库对象（系统表、第三方扩展对象除外）。
- 新增表时必须同步提交表注释与字段注释。
- 新增字段、修改字段语义时，必须同步新增或更新字段中文注释。
- 新增索引、约束时：
  - 若数据库支持对象注释，必须补齐中文注释并说明业务用途。
  - 若数据库不支持对象注释，必须在迁移文件中以同名说明块记录用途。
- 使用 ORM（如 GORM）时，Model 字段必须声明 `comment`；仅在模型中声明不视为完成，必须保证注释实际落库（迁移 SQL、迁移工具或仓储迁移逻辑三选一）。
- 合并前必须执行“缺失注释检查”，结果必须为 0 行；否则禁止合并与发布。
- 历史表整改策略：新增/变更对象必须立即满足；历史存量对象按触达模块逐步补齐，不要求一次性全库改造。

### 数值字段规范（必须执行）
- 涉及金额、积分、费率、折扣、汇率、税率、重量/体积等需保留小数精度的业务字段，数据库类型必须使用 `DECIMAL/NUMERIC(p,s)`。
- 禁止使用 `FLOAT/DOUBLE/REAL` 等浮点类型存储上述业务值。
- Go 代码中对应字段必须使用高精度十进制类型（推荐 `github.com/shopspring/decimal`，或项目统一批准的等价库）；禁止使用 `float32/float64` 承载业务小数。
- 外部接口返回浮点值时，必须在边界层完成精度安全转换后再落库/计算，禁止直接以浮点参与核心业务计算。

## 文件编码与写入规范（必须执行）
- 编码强制要求：所有文件（含历史文件与后续新增文件）统一使用 `UTF-8 无 BOM`；禁止使用 `UTF-8 BOM`、`UTF-16`、`Unicode` 等其他编码。
- 文件禁止带 BOM（`EF BB BF`）；若发现 BOM，除非我明确要求修复，否则不得顺手批量改动其他文件编码。
- 修改现有文件时，必须保持原换行符格式；禁止仅因保存动作导致编码或换行符变化。
- 中文乱码防护（必须执行）：
  - 在 Windows PowerShell/终端读取中文文件时，必须显式指定 UTF-8（例如 `Get-Content -Encoding UTF8`）；禁止省略编码参数直接读取并据此修改中文内容。
  - 终端出现乱码时，禁止直接复制回写；必须先按 UTF-8 重新读取原文件后再编辑。
  - 修改含中文文件后，提交前必须在 IDE diff 中确认中文可读且语义正确；如发现乱码，先修复编码问题再继续开发。
- 修改代码时，优先使用补丁式编辑（如 `apply_patch`）做最小改动，避免整文件读出后重新写回。
- 禁止使用 PowerShell 直接整文件重写源码文件内容，尤其禁止使用 `Set-Content`、`Out-File`、`>`、`>>`、`[System.IO.File]::WriteAllText()` 等可能改变编码的方式修改代码。
- 编辑前应先确认目标文件编码；编辑后必须再次检查文件编码和 BOM 头，确认未引入 `EF BB BF`。
- 如果当前工具无法 100% 保证保留原文件编码和换行符，则必须中断执行，先说明风险并等待我确认。
